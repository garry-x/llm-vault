当前 LLM 分词器：最新模型、BPE 变种与非 BPE 路线

LLM 的 tokenizer 本质上决定了文本如何进入 embedding 表，也决定了上下文、计费和推理长度的实际口径。截至 2026-06-03，最新主流模型包括 GPT-5.5、Claude Opus 4.8、Gemini 3.5、DeepSeek V4、Qwen3.6、Kimi K2.6、MiniMax M2.7 等；它们仍以 BPE 系列或闭源 tokenizer 为主，但实现形态已经分化为 SentencePiece BPE、byte-level BPE、tiktoken BPE、Tekken 和厂商专用编码流程。

一句话结论

如果只看当前通用生成式 LLM，BPE 仍是默认选择；OpenAI、Qwen、DeepSeek、Mistral、Llama 的公开模型基本都落在 BPE 家族。闭源模型如 Claude Opus 4.8 与 Gemini 3.5 则必须以官方 token counting API 为准。

“BPE”不是单一实现。真实 tokenizer 还包括 Unicode 归一化、正则预切分、空格处理、byte fallback、特殊 token、chat template 和多模态占位 token。两个模型同称 BPE，也不能混用 token id。

使用判断

做分词测试时，应优先使用模型随附的 tokenizer 文件或官方 token counting API。对闭源模型，不要用 OpenAI 的 o200k_base 去估算 Claude/Gemini 的真实账单，除非只做粗略对比。

模型绑定 词表不可混用 chat template 会额外加 token 闭源 tokenizer 常不公开

BPE 及其主要变种

BPE 的共同点是从小单位出发，按训练语料中的高频相邻片段逐步合并，最终得到固定词表。差异主要来自初始单位、预切分方式、空格处理和 OOV 兜底策略。

类型 核心特征 常见模型/家族 适用与注意点
经典 BPE / subword BPE 从字符或 Unicode 片段开始训练合并表；常需处理空格、归一化和未知字符。 LLaMA 1/2、早期 Mistral/Mixtral 常见为 SentencePiece BPE 风格；许多开源 LLM 也沿用。 实现简单、推理快；低资源语言和罕见字符可能切得很碎。
Byte-level BPE / BBPE 初始单位是 UTF-8 byte，天然覆盖全部 Unicode，通常不需要 <unk>。 GPT-2/GPT-3 路线、OpenAI tiktoken 基础思想、DeepSeek V3/V4 路线、Qwen3/Qwen3.6 技术报告标注的 BBPE。 鲁棒性强；对中日韩、Indic 等多字节文字，词表训练质量直接影响压缩率。
语言/文字系统定向 BBPE 仍是 byte-level BPE，但训练语料和词表预算刻意面向某些文字系统或领域优化。 2026 年 BrahmicTokenizer-131K 面向 Indic/Brahmic 文字，定位为 o200k_base 的 131K 级替代。 适合作为多语压缩研究对照；不是 GPT/Claude/Gemini 的官方 tokenizer。
tiktoken BPE OpenAI 的高性能 BPE 实现，使用正则预切分、ranked mergeable ranks 和特殊 token 表。 OpenAI GPT-4o、o 系列、GPT-5.x/GPT-5.5 系列通常按 o200k_base 预算；GPT-oss 使用扩展的 o200k_harmony；Llama 3 使用 128K 词表并借鉴 tiktoken 路线。 适合 API 计数和高吞吐编码；不同 encoding 的正则和词表都可能不同。
SentencePiece BPE SentencePiece 框架内的 BPE 模式，直接从原始文本训练，可把空格编码为可见边界符。 LLaMA 1/2、Mistral v1/v2、许多多语开源模型。 语言无关、无需外部分词；注意 SentencePiece 同时也支持 Unigram，不能把 SentencePiece 等同于 BPE。
BPE + byte fallback 主词表无法覆盖时退回 byte token，避免未知字符。 Mistral 7B / Mixtral 的 Hugging Face 文档说明其 tokenizer 带 byte fallback；很多现代 tokenizer 都有类似兜底。 提升鲁棒性，尤其适合代码、emoji、罕见符号；fallback 多时序列会变长。
Tekken Mistral 新 tokenizer，官方说明为基于 tiktoken 的 BPE，训练覆盖 100+ 语言，相比旧 SentencePiece tokenizer 更高效。 Mistral Nemo、Pixtral、Ministral 等 V3-Tekken 系列。 工程上属于 BPE 变种，不是新的基础分词算法；需用 Mistral 官方 tokenizer 包或模型配置。

有没有使用其他类型算法？有

Unigram Language Model

从较大的候选子词集合开始，用概率模型和剪枝得到目标词表，编码时可用 Viterbi 选择最可能切分。

SentencePiece T5 / mT5 Gemma 常见归类

WordPiece

与 BPE 类似也是子词词表，但训练目标更偏似然/概率增益，典型符号是 ## continuation。

BERT DeBERTa 早期 encoder

字节/字符级与 tokenizer-free

绕开固定子词词表，直接处理 UTF-8 byte 或 Unicode character，或把 byte 动态聚合为 patch。

ByT5 CANINE BLT

同叫 BPE，为什么不能混用？

1. 归一化

NFKC、大小写、特殊空白和控制字符处理不同，会改变输入序列。

2. 预切分

tiktoken 使用正则把文本先切成片段；SentencePiece 更强调 raw text 训练。

3. 初始单位

字符、Unicode 片段、UTF-8 byte 的选择决定 OOV 行为和多语压缩率。

4. 合并表

merge rank 来自训练语料；中文、代码、数学、emoji 的覆盖取决于语料分布。

5. 特殊 token

BOS/EOS、role、tool、image、audio、reasoning token 会改变真实上下文消耗。

测试与选型结论

用于本仓库后续分词测试时：同一文本应分别用每个模型自己的 tokenizer 编码；记录纯文本 token 数、套用 chat template 后的 token 数，以及 reasoning/tool/multimodal wrapper 的额外开销。DeepSeek V4 这类模型还要按仓库 encoding 脚本生成 prompt 后再编码。BPE 系列可作为主线，Unigram、WordPiece、byte-level/tokenizer-free 可作为对照组。
