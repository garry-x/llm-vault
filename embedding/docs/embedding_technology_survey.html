头部大模型所用 Embedding 与表示技术深度调研

修订日期：2026-06-03 范围：GPT-4o、Gemini、Claude、Llama、DeepSeek、Qwen、Gemma 等头部模型 重点：模型内部表示，而非专用检索 embedding 模型排行榜
核心结论：头部模型中的 embedding 技术，已经从“词表 id 查表得到 token 向量”扩展为一整套表示系统： 文本 token embedding、RoPE/多维 RoPE/时间对齐 RoPE、长上下文位置缩放、视觉 patch embedding、音频 codec 或声学特征 embedding、 视频时空 embedding、跨模态 projector、early fusion 统一 token 序列、MLA 的低秩 KV latent 表示、MoE router 表示，以及 外围 RAG 检索 embedding。专用 embedding 模型只是这套系统的一部分，不应和头部生成模型内部的 embedding 技术混为一谈。

1. 先澄清：这里的 Embedding 指什么

在大模型语境里，“embedding”至少有四层含义。上一版文档主要写的是第四层，因此偏离了目标。

层级 含义 在头部模型中的作用
输入 token embedding 把文本 token id 映射为连续向量，通常与词表、tokenizer、输出 head 绑定。 语言模型的第一层语义表示，是后续 attention/MLP/MoE 的输入。
位置 embedding/位置编码 给 token 注入顺序、距离、二维空间、视频时间戳等位置信息。 决定长上下文、视觉布局、音视频同步、代码/表格结构理解能力。
跨模态 embedding 把图像 patch、音频帧、视频帧、OCR/布局信息投影到 LLM 可消费的 token 空间。 GPT-4o、Gemini、Llama 4、Qwen-Omni/VL 等原生或半原生多模态模型的关键。
检索 embedding 把 query/document 编成向量，用于 RAG、语义搜索、rerank。 多数情况下是外围系统能力，不是生成模型内部参数的一部分。

2. 头部模型采用的表示技术总览

模型/系列 公开可确认的表示相关技术 对 embedding/表示系统的意义
DeepSeek-V3/R1 DeepSeekMoE、Multi-head Latent Attention (MLA)、RoPE 解耦、Multi-Token Prediction、FP8 训练。 重点不是检索 embedding，而是把 K/V 压缩为 latent 表示，降低长上下文推理时 KV cache 成本。
Llama 4 原生多模态、early fusion、文本与视觉 token 进入统一 backbone、MoE；Meta 还提到部分架构不再依赖传统位置 embedding。 代表开源权重模型从“外接视觉 encoder”转向更深层的统一 token 表示。
GPT-4 / GPT-4V GPT-4 技术报告确认其为可接受图像和文本输入、输出文本的 multimodal model；GPT-4V 是 GPT-4 视觉能力集成项目。 公开资料证明 GPT 系列进入图文多模态阶段，但 GPT-4 报告明确不披露模型大小、具体架构、训练 compute 等细节。
GPT-4o / GPT-4.1 / GPT-5 GPT-4o 是端到端 text/vision/audio omni 模型；4o 原生图像生成是自回归并深度嵌入 omnimodal GPT-4o；GPT-4.1 支持 1M context；GPT-5 是带实时 router 的统一系统。 GPT 公开资料最有价值的是“多模态统一网络、原生生成、长上下文、系统级路由”的能力信号，而非可复现的 embedding 层细节。
Gemini 系列 原生多模态、长上下文、多语言/代码能力；Gemini Embedding 报告也说明 Gemini backbone 可迁移到通用表示任务。 体现 foundation model 表示能力可以服务生成、检索、分类、聚类等多种下游任务。
Qwen2.5-VL / Qwen2.5-Omni / Qwen3 MRoPE、多分辨率视觉位置编码、动态分辨率、动态 FPS；Omni 中有 TMRoPE 以对齐音视频时间。 Qwen 路线强调把空间、时间、多分辨率信息编码进可被 LLM 处理的位置表示。
Claude 3/3.5/4 公开资料确认强视觉、长上下文和多模态能力，但 Anthropic 很少披露底层 embedding 架构细节。 可作为能力参照，但不适合作为内部 embedding 技术细节的主要证据来源。
Gemma 3 开源实现显示其 vision-language backbone 会把图像 embedding 放入文本 embedding 序列对应位置；使用 RoPE 等机制。 适合观察开放模型中“视觉 encoder + projector + LLM 序列融合”的工程实现。

3. GPT 系列公开资料能补充什么

OpenAI 对 GPT 系列的底层 embedding 细节披露很克制。GPT-4 技术报告确认 GPT-4 是大规模、多模态模型， 可接受图像和文本输入并输出文本；但报告也明确不提供进一步的架构、模型大小、硬件、训练 compute、数据构造和训练方法细节。 因此，对 GPT 的调研应把“官方确认的能力和系统描述”与“社区架构猜测”分开。

公开资料 可确认内容 对 embedding/表示技术的启示
GPT-4 Technical Report / GPT-4V GPT-4 是图文输入、文本输出的 multimodal model；GPT-4V 文档说明视觉能力是 GPT-4 多模态集成工作的一部分。 可以确认 GPT 进入图像+文本输入表示阶段，但不能确认视觉 encoder、projector、位置编码或融合层细节。
Hello GPT-4o / GPT-4o System Card GPT-4o 可接受文本、音频、图像、视频组合输入，并生成文本、音频、图像输出；OpenAI 称其端到端跨文本、视觉、音频训练，输入输出由同一个神经网络处理。 这是 GPT 系列公开资料中最强的“统一多模态表示”证据：相比旧 Voice Mode 的 ASR→LLM→TTS 管线，4o 把音色、说话人、背景声等信息直接交给主模型处理。
GPT-4o Native Image Generation Addendum 4o 图像生成不同于 DALL-E 扩散模型，是自回归模型，并且原生嵌入 GPT-4o 的 omnimodal 架构深处。 这说明图像不只是外部工具输出，而是进入 GPT-4o 内部生成表示系统；文本渲染、图像编辑和多图变换都受主模型知识与指令理解影响。
GPT-4.1 发布资料 GPT-4.1/mini/nano 支持最高 1M tokens context，并在长上下文、多模态视频理解、代码和指令跟随上提升。 虽然未披露位置编码细节，但可以确认 GPT 路线在长上下文表示上投入明显，重点是“跨 1M 上下文可靠注意到相关信息、忽略干扰”。
GPT-5 System Card GPT-5 被描述为统一系统，包含 fast/main 模型、thinking 模型和实时 router；router 根据复杂度、工具需求和用户意图选择模型。 这不是 embedding 层细节，但体现了头部模型从“单模型表示”走向“系统级表示与路由”：输入会先被系统判断任务类型，再进入不同能力路径。
写 GPT 时应避免使用未证实说法，例如具体 MoE 专家数、参数量、RoPE 方案、视觉 projector 结构等。 这些在公开官方资料中没有被确认，最多只能作为“社区传闻/推测”，不应写入正式技术调研的事实部分。

4. 文本 Token Embedding：从词表到 tied embedding

对 decoder-only LLM，第一步仍然是把离散 token id 通过 embedding matrix 映射为 hidden vector。 现代头部模型在这一层的重点不是“有没有 embedding”，而是词表设计、特殊 token、代码/数学/多语种覆盖、输入输出权重共享、 以及 embedding 层在并行训练中的放置。

DeepSeek-V3 技术报告和相关实现分析提到，其浅层结构包含 embedding layer，深层包含 output head；大规模流水并行中， embedding 与 output head 的放置会影响通信和负载。对 Qwen、Llama、Gemma 等开放模型，也可以从 tokenizer 与 config 看到词表、特殊 token、embedding size、tie word embeddings 等设计。

实践含义：tokenizer 是 embedding 技术的一部分。中文、代码、数学公式、表格、JSON、工具调用 schema 如果被切得很碎， 不仅增加上下文长度，还会影响早期 token embedding 的统计质量和后续 attention 成本。

5. 位置表示：RoPE 是当前主流，变体越来越重要

绝对位置 embedding 在早期 Transformer 中常见，但头部 LLM 普遍转向 RoPE 或其变体。RoPE 通过旋转 query/key 的子空间来注入相对位置信息，适合自回归 attention，也便于长上下文外推和缩放。

RoPE

文本顺序相对位置

用于多数 decoder-only LLM。长上下文扩展通常围绕 RoPE base、scale、插值或再训练展开。

MRoPE

二维空间多分辨率

Qwen-VL 路线用多维/多分辨率 RoPE 编码图像 patch 的空间位置和视频帧结构。

TMRoPE

音视频同步时间戳

Qwen2.5-Omni 报告提出 Time-aligned Multimodal RoPE，用于对齐音频与视频输入时间。

长上下文能力并不只是“把最大 token 数调大”。位置表示必须让模型在训练长度外仍能理解距离关系；同时 KV cache、 attention 复杂度、检索增强和上下文压缩都会成为瓶颈。因此 DeepSeek 的 MLA、Llama 的长上下文设计、Qwen2.5-1M 的长上下文训练，都应被视为表示系统的一部分。

6. DeepSeek 的关键点：MLA 是 Latent 表示压缩

DeepSeek-V3 最值得写进本调研的不是“有没有 embedding API”，而是 MLA 对注意力表示的重构。 标准 MHA/GQA 在推理时需要缓存每层每个 token 的 K/V；上下文越长，KV cache 越大。MLA 通过低秩投影把 K/V 压缩为更小的 latent vector，并把带 RoPE 的位置部分与内容部分解耦。

这个设计的本质是：不是所有注意力所需信息都必须以完整 K/V 张量形式缓存。内容匹配可进入压缩 latent 空间， 位置敏感部分则保留独立的 RoPE component。这样可以显著降低长上下文推理时的显存压力，也使头部模型的“内部表示” 从简单 hidden state 发展为多路、低秩、可缓存的表示结构。

7. 多模态 Embedding：从外接 Encoder 到统一 Token 序列

多模态大模型的 embedding 技术可以分成三代：

  1. 外接 encoder + projector：图像由 CLIP/SigLIP/ViT 编成 patch embedding，经 projector 映射到 LLM hidden size，再插入文本序列。
  2. 更紧密的 early fusion：文本 token 与视觉 token 在更早阶段进入统一 backbone，Meta 对 Llama 4 的描述属于这一方向。
  3. omni token 系统：文本、图像、音频、视频都被 token 化并进入同一自回归或统一建模框架，GPT-4o、Qwen-Omni、Gemini 系列体现这个趋势。

视觉 embedding 不只是把图片压成一个向量。文档理解、图表推理、GUI 操作和视频问答都要求模型保留 patch 级细节、 二维布局、分辨率、帧时间、OCR 文本位置和跨帧事件。Qwen2.5-VL 的动态分辨率与动态 FPS、Qwen2.5-Omni 的 TMRoPE、 Llama 4 的 early fusion，都在解决“不同模态如何进入同一个语义和位置空间”的问题。

8. 音频与视频：Embedding 进入时序建模

音频模型通常要把 waveform 转成声学 token、codec token 或 spectrogram-like 表示；视频则要把帧、patch、时间戳、 语音转写和画面事件对齐。GPT-4o 被描述为可接受文本、音频、图像、视频组合输入并生成多种输出，Qwen2.5-Omni 公开提出音视频交错组织和 TMRoPE。它们共同说明：头部模型的 embedding 技术正在从“静态 token 表”走向 “多模态时空 token 流”。

工程上，音视频 embedding 的难点不是单帧理解，而是 token budget。1 小时视频若按高 FPS 和高分辨率直接展开， token 数不可接受。因此头部模型会结合动态采样、分辨率压缩、事件片段切分、音频转写、关键帧抽取和时间位置编码。

9. MoE Router 与专家表示

MoE 本身不是 embedding 层，但它改变了 hidden representation 的流动方式。DeepSeekMoE、Llama 4 MoE、Qwen3 MoE 等模型会用 router 根据 token hidden state 选择专家。也就是说，token embedding 经若干层后形成的表示不仅用于 预测下一个 token，也用于决定“交给哪些专家处理”。

这带来两个表示层面的变化：第一，模型容量被分散到专家子空间，不同 token/领域/语言可能激活不同路径； 第二，router 表示质量会影响负载均衡、专家专业化和推理稳定性。因此 MoE 头部模型通常配合负载均衡策略、 专家共享、auxiliary-loss-free balancing 或全局 batch balancing。

10. 检索 Embedding：外围能力，但与头部模型协同

专用 embedding 模型仍然重要，但它们通常属于 RAG 系统外围，而不是 GPT-4o/Claude/DeepSeek 这类生成模型的内部 embedding 层。常见组合是：用 Qwen3-Embedding、BGE-M3、Jina、Voyage、Cohere、OpenAI embedding 等模型召回文档， 再把检索结果交给 DeepSeek、Claude、Gemini、GPT、Qwen 或 Llama 生成答案。

头部生成模型和检索 embedding 的关系正在变深：强 LLM 可用于合成 query/document 训练数据、构造 hard negatives、 蒸馏 reranker 偏好、给 chunk 补全上下文；而检索 embedding 则为长上下文模型降低输入负担，减少无关 token。

11. 关键技术趋势

趋势 代表技术 为什么重要
位置编码从一维走向多维时空 RoPE、MRoPE、TMRoPE、长上下文 RoPE scaling 文本、图像、音频、视频需要共享序列，但位置语义完全不同。
多模态 token 早融合 Llama 4 early fusion、GPT-4o omni、Qwen-Omni 减少“视觉理解是外挂”的割裂，让跨模态推理发生在主干模型内部。
KV cache 表示压缩 DeepSeek MLA、低秩 K/V latent、RoPE 解耦 长上下文和高并发推理的核心成本在缓存，而不只在参数量。
表示驱动专家路由 DeepSeekMoE、Llama 4 MoE、Qwen MoE hidden state 同时承担语义预测和计算路径选择。
检索向量与生成模型协同训练 LLM 合成数据、reranker 蒸馏、contextual retrieval RAG 的质量来自检索表示与生成模型上下文理解的共同优化。

12. 给工程团队的判断框架

  1. 如果你问“模型内部怎么理解输入”：重点看 tokenizer、token embedding、RoPE/MRoPE、视觉/audio projector、early fusion、KV cache 表示。
  2. 如果你问“RAG 怎么召回文档”：重点看专用 embedding 模型、chunk 策略、向量维度、reranker 和混合检索。
  3. 如果你问“长上下文为什么贵”：重点看 attention 复杂度、KV cache、MLA/GQA/MQA、RoPE scaling 和上下文压缩。
  4. 如果你问“多模态为什么难”：重点看不同模态 token 的粒度、位置系统、时间同步、projector 训练和 token budget。
