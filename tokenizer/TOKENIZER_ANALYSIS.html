比较口径

文本为语义对应的原创微型小说中英文版本。结果适合在同一语言内横向比较 tokenizer，而不是把中英文绝对数直接解释为语言能力。

压缩率 = 字符数 / token 数

数值越高，表示每个 token 覆盖的原文字符越多，即同样上下文预算能承载更多文本。

先看结论

  • 中文Kimi K2.6 为 1.64 chars/token，MiniMax M2.7 为 1.57，领先本次公开验证集合。
  • 英文14 个本地结果集中在 224–245 tokens，分化远小于中文。
  • 更新Qwen3.6 的中文结果为 256 tokens，相比此前 Qwen3 8B 代表的 268 明显变化。
  • 边界GPT-5.5、Claude Opus 4.7、Gemini 3.1 走官方 API；公开本地基线不得冒充最新服务模型。

模型家族解读

OpenAI

GPT-5.5 通过官方 responses/input_tokens 精确计数；o200k_base 只作为公开本地切片基线。

Claude / Gemini

当前精确路径是官方计数 API。它们与本地纯文本编码的口径不同，不假设存在可下载切片。

中文公开模型

Kimi K2.6、DeepSeek V4 Pro、Qwen3.6、Hy3、MiniMax M2.7 等均有官方 tokenizer 工件；本样本由 Kimi 领先。

Mistral / NVIDIA

已升级到 Mistral Medium 3.5 与 Nemotron 3 Super；两者在本中文样本均为 376 tokens。

xAI Grok

Grok 4.3 的官方 /v1/tokenize-text 可返回切片；Grok-2 下载工件仅作为无需密钥的基线。

获取边界

Llama 4 需要仓库许可；其他已验证新版本均直接读取公开文件，无需执行远程 tokenizer 代码。

实践建议：成本估算应固定实际模型 id，并使用业务真实语料分别测试中文、英文与混合文本。对 RAG 或长上下文服务，中文 tokenizer 的 20%–30% token 差异可直接转化为容量与费用差异。
