比较口径

文本为语义对应的原创微型小说中英文版本。结果适合在同一语言内横向比较 tokenizer，而不是把中英文绝对数直接解释为语言能力。

压缩率 = 字符数 / token 数

数值越高，表示每个 token 覆盖的原文字符越多，即同样上下文预算能承载更多文本。

先看结论

  • 中文DeepSeek V3.2 与 V4 Pro 并列第一，Seed、GLM 紧随其后，均达到 1.45 chars/token 以上。
  • 英文10 个 tokenizer 集中在 224–245 tokens，分化远小于中文。
  • 注意词表越大不必然中文越省 token：Cohere 约 255k 词表仍弱于 DeepSeek 约 128k。
  • 边界Claude 与 Gemini 只有官方 API 计数路径，不能用第三方切片替代线上真实 tokenizer。

chars/token，越高越好；英文差距相对收敛

模型家族解读

OpenAI

tiktoken 为官方开源实现；GPT-4o 的 o200k_base 可本地查看 id 与字节片段。

Claude / Gemini

当前精确路径是官方计数 API。它们与本地纯文本编码的口径不同，不应假设可视化切片。

中文公开模型

DeepSeek、Qwen、GLM、ERNIE、Seed 均发布 tokenizer 资源。本样本中 DeepSeek V3.2 与 V4 Pro 的 token id 序列相同，但 V4 Pro 词表多 465 个条目。

Mistral

官方维护 mistral-common；本例对所选仓库启用了 Transformers 提示的 regex 修正。

xAI Grok

官方发布 tokenizer.tok.json，为 tiktoken 格式；本程序复现 SGLang 公开解析路径。

授权与信任边界

执行 remote code 或接受模型许可，是获取 tokenizer 的边界，不代表相应模型缺少 tokenizer。

实践建议：成本估算应固定实际模型 id，并使用业务真实语料分别测试中文、英文与混合文本。对 RAG 或长上下文服务，中文 tokenizer 的 20%–30% token 差异可直接转化为容量与费用差异。
